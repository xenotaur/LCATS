# Evidence Record: EV-0003

## Method

EV-0002 (2026-07-15) validated the special-character pipeline via a
**deterministic simulated regeneration**: `normalize_story_dict` applied
in memory to the existing `data/` tree, not a literal `lcats gather`. Its
own Confidence section named the remaining gap explicitly: "The literal
full `data/` regeneration + `lcats survey --mode specials` run remains a
maintainer operational step to confirm end-to-end."

This record closes that gap. The maintainer ran
`lcats/docs/reference/prepare-corpora-release.md` (WI-RELEASE-0021) as a
literal, human-executed runbook — no agent involvement, real network
`lcats gather`, real `lcats clean`, real CLI commands — per the runbook's
own independent-verification rationale (agent-run checks are only as good
as what the agent was told to check; matches standard IV&V practice).

The dogfood run itself surfaced and fixed five real defects along the way
(each tracked in its own PR and execution record under
`project/executions/`, not restated here): a dangling-symlink crash
in the release runbook's clear-step (PR #128), two residual
`review_needed` findings needing per-story overrides (PR #129), a
dangling-symlink crash in the underlying gather pipeline itself plus a
`lcats clean` command to fix it for good (WI-CLEAN-0022, PRs #130-131),
and a doc-accuracy gap found on a final re-verification pass (PR #132).
This record is the payoff of that process: the pipeline, as hardened by
those fixes, produces a genuinely clean corpus on a real run.

## Observed Signals

- `lcats survey --mode specials data/ --no-progress` (step 4 of the
  runbook, after a literal clear + full network `lcats gather`): **no
  output, exit code 0** — confirmed directly by the maintainer.
- `lcats repair-specials data/mass_quantities/deny_the_slake__wilson.json
  --format jsonl` (step 5): **empty output**. This file was EV-0002's own
  historical example of a repaired mojibake occurrence (`Ã©`); empty
  output here means the gather-time repair rule already fixed it before
  the file was ever written to disk, consistent with EV-0002's prediction.
- `lcats promote --dry-run` (step 6): **`would promote:` for all 12
  collections, zero `blocked:` lines** —
  `anderson`, `chesterton`, `grimm`, `hemingway`, `london`, `lovecraft`,
  `mass_quantities`, `ohenry-four_million`, `ohenry-whirligigs`,
  `sherlock`, `wilde_happy_prince`, `wodehouse`. `lcats promote`'s
  internal survey gate (`promote.survey_collection()`,
  `lcats/lcats/analysis/corpus/promote.py:62-112`) calls the same
  `specials.iter_special_characters`/`classify_character` code path the
  CLI `survey` command does, so this is not two independent
  implementations agreeing — a bug in that shared classifier or the
  allowlist would affect both equally. The two callers do differ in one
  respect: the CLI applies the legacy `unicode.DEFAULT_EXCLUDED_CHARS`
  exclusion list, while `promote`'s check passes an empty exclusion set,
  and this pair has previously disagreed for exactly that reason (a
  finding suppressed by the CLI's exclusion list but caught by
  `promote`'s stricter check). Their agreement here rules out that
  specific, previously-observed discrepancy; it does not rule out a bug
  in the shared classification logic itself.

## Interpretation

The special-character repair pipeline (measured rule table +
gather-time normalization hook + per-story overrides + allowlist) works
on a real, literal, non-simulated corpus regeneration, not just in
simulation. This is the "hard data" step of the maintainer's original
plan: (a) produce a dogfood runbook, (b) shake out bugs via a real human
running it, (c) confirm the system works, (d) [assess re-run, deferred —
separate activity, real API cost], (e) revisit the span-op/review/apply
dead-code question with this evidence in hand (see the corresponding
2026-07-18 decision log entry).

## Confidence

- High confidence: `lcats survey`'s CLI and `lcats promote`'s internal
  check agree the corpus is clean on a real, literal regeneration, not
  a simulation. As noted above, both share the same underlying
  classifier, so this rules out the known exclusion-list discrepancy
  between the two callers, not a shared-classifier defect — a genuinely
  independent check would need a second, differently-implemented
  classifier, which does not exist today.
- This directly satisfies WS-SPECIALS-CLEANUP's first exit criterion
  ("`lcats survey --mode specials` over freshly regenerated `data/`
  reports zero unaddressed mojibake findings") with real, not simulated,
  evidence.
